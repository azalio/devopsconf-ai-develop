# Как разрабатывать production Kubernetes-операторы с помощью LLM так, чтобы это можно было повторить у себя

Ниже — полная переписанная версия презентации под цель:

**не просто показать кейс, а дать слушателям воспроизводимый процесс**.

Главная ось всей презентации:

**SPEC → PLAN → TEST → CODE → REVIEW → LEARN**

Главная мысль:

**LLM ускоряет не сам по себе. Он ускоряет только тогда, когда встроен в инженерный процесс с явными артефактами, ограничениями и проверками.**

## Слайд 2. Для кого этот доклад

- Для тех, кто делает internal platform tooling
- Для тех, кто пишет контроллеры, admission webhooks и операторов
- Для тех, кто уже пробовал LLM в разработке и получил смешанные результаты
- Для тех, кто хочет не «магии», а воспроизводимого процесса

**Что сказать голосом:**

Если вы один раз попросили AI «сделай мне контроллер» и получили красивый, но опасный код — этот доклад для вас. Если вы хотите встроить LLM в реальную инженерную работу, а не в демо — тоже.

## Слайд 3. Главная проблема

- Почти все уже попробовали AI в разработке
- Но результаты очень неровные
- На простых задачах кажется, что всё ускорилось
- На сложных системах часто приходит хаос:
  - потеря модели
  - большие diff
  - неверные архитектурные решения
  - опасные допущения
  - ложное ощущение скорости

**Ключевой тезис:**

**Проблема не в том, что LLM плохо пишет код. Проблема в том, что большинство встраивает его в плохой процесс.**

## Слайд 4. О чём этот доклад

- Почему AI на сложной backend-задаче может и ускорять, и замедлять
- Почему Kubernetes-оператор — хороший стресс-тест для LLM-подхода
- Реальный кейс: `Project Operator`
- Инженерный цикл из 6 шагов:
  - `SPEC`
  - `PLAN`
  - `TEST`
  - `CODE`
  - `REVIEW`
  - `LEARN`
- Что именно делать на каждом шаге
- Какие артефакты должны получаться на выходе
- Какие анти-паттерны ломают весь процесс

**Что сказать голосом:**

Главный результат этого доклада — не вдохновение, а рабочая схема, которую вы сможете унести в свой проект.

## Слайд 5. Реальный кейс: Project Operator

Мы не делали игрушечный CRUD-controller.

Мы делали `Project Operator`, который:

- объединяет несколько namespace в один логический проект
- задаёт общие project-level quotas
- выдаёт доступы на уровне проекта
- разводит роли cluster admin и project admin
- синхронизирует desired state во всех namespace проекта

Основные ресурсы:

- `Project`
- `ProjectRole`
- `ProjectAccessBinding`

**Что сказать голосом:**

Это важный момент. Я специально показываю не trivial demo, а задачу, где есть модель владения, quota safety, access model, reconciliation, webhook validation и тестирование.

---

## Слайд 6. Почему именно оператор — хорошая проверка метода

Потому что здесь одновременно есть:

- доменная модель
- CRD schema design
- reconcile loop
- webhook validation
- cross-namespace ownership
- RBAC
- finalizers
- envtest / integration tests
- edge cases и partial failures

**Основная сложность:**

Не в том, чтобы написать 200 строк Go-кода.  
А в том, чтобы **удержать модель целиком**.

## Слайд 7. Что должен унести слушатель

После доклада человек должен уметь:

- понять, когда LLM вообще стоит использовать на backend/system design задаче
- начать не с генерации кода, а со спецификации
- разбить большую задачу на атомарные контракты
- заставить AI генерировать маленькие проверяемые куски
- отделить написание кода от написания тестов и review
- сохранять знания проекта так, чтобы следующая сессия была сильнее предыдущей

**Что сказать голосом:**

Это не доклад про промпты как магические заклинания. Это доклад про управляемый цикл разработки.

## Слайд 8. Наивный подход, который ломает всё

Типичный запрос:

> Мне нужен Kubernetes-оператор, который объединяет namespace'ы в проекты, задаёт общие квоты и управляет доступом. Сделай.

Что происходит дальше:

- AI сам принимает архитектурные решения за вас
- сам придумывает quota model
- сам упрощает access model
- генерирует слишком большой diff
- смешивает бизнес-логику и побочные эффекты
- не удерживает edge cases
- пишет тесты под свою же реализацию
- создаёт опасную иллюзию прогресса

**Итог:**

Код есть, контроля нет.

## Слайд 9. Ключевой инсайт

**LLM — не автопилот. LLM — второй пилот.**

Разница между провалом и ускорением не в модели.  
Разница в процессе вокруг неё.

Плохой поток:

`idea → prompt → code → hope`

Хороший поток:

`SPEC → PLAN → TEST → CODE → REVIEW → LEARN`

**Что сказать голосом:**

Как только вы превращаете разработку в цепочку явных инженерных артефактов, LLM начинает помогать. Пока этого нет, он усиливает хаос.

---

## Слайд 10. Главная схема доклада

**SPEC → PLAN → TEST → CODE → REVIEW → LEARN**

Это **рекомендуемая стартовая последовательность**.
В реальной работе — **итеративный цикл** с возвратами между этапами.

Для каждого этапа есть:

- вход
- действие
- выход
- критерий качества
- типичные ошибки

**Что сказать голосом:**

Возвраты между этапами — норма, а не ошибка. Из CODE вы часто возвращаетесь в SPEC, когда обнаруживаете недосказанное. Из TEST — в PLAN, когда понимаете, что задача разбита неправильно. Но без этой опорной последовательности AI быстро размывает процесс.

## Слайд 11. Как читать остальные слайды

На каждом из следующих этапов я покажу 5 вещей:

1. Что является входом
2. Что делает человек
3. Чем помогает LLM
4. Какой артефакт должен появиться на выходе
5. Как понять, что шаг сделан достаточно хорошо

## Слайд 12. SPEC — зачем он нужен

**Input:**

- бизнес-задача
- ограничения
- существующая архитектура
- ваши реальные инварианты

**Задача этапа:**

Не дать AI принимать продуктовые и архитектурные решения за вас.

**Выход этапа:**

- spec-док
- список инвариантов
- список edge cases
- acceptance criteria
- список out-of-scope

**Главная мысль:**

Без spec AI решает за вас.  
Со spec AI реализует ваши решения.

---

## Слайд 13. SPEC на примере Project Operator

В spec для `Project Operator` должны быть зафиксированы вещи вроде:

- `Project` объединяет несколько namespace в одну логическую сущность
- каждый namespace принадлежит ровно одному проекту
- project quota задаётся сверху и не должна нарушаться суммарно
- namespace attach происходит через label
- project admin живёт внутри ограничений, заданных cluster admin
- `ProjectAccessBinding` — основной user-facing способ выдачи доступа
- `ProjectRole` нужен только для кастомных ролей
- уменьшение project quota ниже текущей утилизации допустимо, но новые нарушения должны блокироваться отдельно

**Что сказать голосом:**

Заметьте: всё это — не код. Но именно это определяет, будет ваш оператор корректным или нет.

---

## Слайд 14. Шаблон хорошего SPEC

Структура, которая работает:

- Цель
- Модель ресурсов
- Кто что может
- Инварианты
- Edge cases
- Failure modes
- Out of scope
- Acceptance criteria

**Пример инвариантов:**

- Every namespace MUST belong to exactly one `Project`
- Total project resource consumption MUST respect project limits
- Project admin MUST NOT be able to change project quota
- Reconciler MUST be idempotent

---

## Слайд 15. Практический prompt для SPEC

```text
Мне нужен Kubernetes Project Operator.

Ресурсы:
- Project
- ProjectRole
- ProjectAccessBinding

Project объединяет namespace'ы в одну логическую сущность.
Namespace присоединяются к проекту через label.
У проекта есть общая квота CPU / memory / storage.
ProjectAccessBinding выдаёт доступ к проекту.
ProjectRole нужен только для кастомных ролей.
Cluster admin задаёт границы.
Project admin работает только внутри них.

Задай вопросы, которые нужны для написания полноценного spec.
Не предлагай код. Сначала только модель, инварианты, failure modes и edge cases.
```

**Что сказать голосом:**

В этот момент AI полезен как аналитик: он помогает дожать недосказанные решения. Но финальные ответы даёт инженер.

---

## Слайд 16. Devil’s Advocate после SPEC

После первого spec полезно добивать так:

```text
Выступи как критик этой модели.
Найди:
- race conditions
- ownership ambiguity
- quota safety issues
- partial failures
- delete/update edge cases
- authorization boundary mistakes

Сформулируй, каких решений в spec не хватает.
```

**Усиление:** для критичных архитектурных решений полезно получить независимые мнения от разных моделей ([llm-council](https://github.com/karpathy/llm-council)), а не просить одну модель критиковать саму себя.

**Выход:**

Обновлённый spec, в котором уже меньше сюрпризов на этапе кода.

---

## Слайд 17. Критерий качества для SPEC

SPEC считается достаточным, если:

- два инженера одинаково понимают desired behavior
- AI больше не придумывает модель сам
- по spec можно написать тесты до реализации
- основные edge cases уже зафиксированы

**Красный флаг:**

Если после чтения spec вы всё ещё не понимаете, что система должна делать при partial failure, delete, quota decrease или ownership conflict — spec ещё не готов.

---

## Слайд 18. PLAN — зачем он нужен

**Input:**

- утверждённый spec

**Задача этапа:**

Разбить систему не на «большой оператор», а на последовательность маленьких инженерных задач.

**Выход этапа:**

- implementation plan
- очередность задач
- зависимости
- список файлов / пакетов / компонентов
- критерии готовности каждого шага

**Главная мысль:**

Большая задача почти всегда ломает LLM.  
Маленькая задача с явным контрактом — почти всегда помогает.

---

## Слайд 19. Задача + критерий готовности

Каждая подзадача формулируется так:

**Что сделать → Как понять, что сделано**

Примеры:

- Определить CRD types → `make generate` проходит без ошибок
- Namespace attachment → namespace'ы с label проекта появляются в `status.namespaces`
- Quota enforcement → webhook отклоняет pod, если суммарное потребление превышает `project.spec.quotas`
- Access reconciliation → RoleBinding существует в каждом namespace проекта для каждого ProjectAccessBinding
- Namespace validation webhook → namespace с label несуществующего проекта отклоняется

**Что сказать голосом:**

Критерий готовности — это главное. Без него задача размыта, и LLM будет додумывать за вас. С ним — у вас есть конкретная проверка, прошёл шаг или нет.

---

## Слайд 20. PLAN на примере Project Operator

Разумная последовательность шагов:

1. Scaffold проекта (kubebuilder init, cert-manager в kind)
2. Определить CRD types
3. Прогнать `make generate`
4–9. Для каждой задачи (namespace attachment, quota, access, webhooks, status):
   - сначала тесты (чистая сессия)
   - потом реализация (другая сессия)
10. Проверить покрытие, добить интеграционные тесты
11. Прогнать review и cleanup
12. Прогнать e2e / kind

**Что сказать голосом:**

Обратите внимание: тесты пишутся не в конце одним блоком, а перед каждой задачей. И обязательно в отдельной сессии — иначе AI пишет тесты под свой же код, а не под spec.

---

## Слайд 21. Pre/Post Conditions как контракт

Для каждой подзадачи полезно задавать:

**PRE:**

- какие сущности уже существуют
- какие assumptions допустимы
- что должно быть подготовлено заранее

**POST:**

- что станет истинно после выполнения шага
- что можно проверять тестом

**Пример:**

PRE:
- `Project` существует
- namespace'ы разрешимы
- CRD types уже сгенерированы

POST:
- все namespace с label проекта прикреплены к проекту
- суммарные quotas не превышают project limits
- access bindings отражают desired state

## Слайд 22. Практический prompt для PLAN

```text
У меня есть утверждённый spec для Project Operator.
Разбей реализацию на маленькие задачи.

Для каждой задачи укажи:
- имя
- цель
- зависимости
- PRE conditions
- POST conditions
- как проверить завершённость

Ограничения:
- одна задача = один логический шаг
- не смешивай CRD, reconciler, webhook и tests в одном большом шаге
```

## Слайд 23. Критерий качества для PLAN

PLAN считается хорошим, если:

- каждую задачу можно дать AI отдельно
- каждую задачу можно проверить независимо
- зависимости понятны заранее
- нет больших зон «сейчас просто сделаем всё остальное»

**Красный флаг:**

Если ваш план заканчивается пунктом «ну а дальше реализовать контроллер», это не план, а иллюзия плана.

## Слайд 24. TEST — почему тесты идут до кода

**Input:**

- spec
- implementation plan
- конкретная задача из PLAN
- acceptance criteria

**Задача этапа:**

Превратить требования и инварианты в проверяемый контракт **до** написания кода.

**Выход:**

- unit / envtest / integration tests
- тесты компилируются, но падают (реализации ещё нет)
- по тестам можно понять expected behavior системы

**Главная мысль:**

Если код уже в контексте, AI пишет тесты под реализацию, включая её баги. Тесты пишутся в чистой сессии — без контекста кода.

---

## Слайд 25. Правильный поток TDD с AI

Неправильно:

- «Напиши код и сразу тесты к нему»

Правильно:

1. В чистой сессии попросить тесты по SPEC и PLAN
2. Человеку прочитать тесты
3. Закоммитить тесты как контракт
4. В другой сессии просить реализацию, которая проходит эти тесты

Формула:

**tests first, implementation second, разные сессии**

**Практический нюанс:**

На практике порядок может быть и TEST→CODE, и CODE→TEST. Главное правило — **разные сессии**. Тот, кто пишет тесты, не должен видеть реализацию в контексте.

- TEST→CODE — безопаснее для критичных мест (webhooks, quota, access control)
- CODE→TEST — быстрее, когда спек чёткий и задача понятна

---

## Слайд 26. Практический prompt для TEST

```text
Напиши envtest тесты для задачи 7 из PLAN.md.
Сценарии бери из acceptance criteria в SPEC.md.
Только тесты. Не реализуй reconciler.
Тесты должны падать.
```

## Слайд 27. Что именно тестировать у операторов

Минимальный набор:

- happy path
- idempotency
- partial failures
- update path
- delete path
- authorization boundaries
- admission validation
- edge cases бизнес-модели

**Что сказать голосом:**

Одна из самых частых ошибок — написать красивые тесты на моки, которые почти ничего не говорят о корректности reconcile-поведения.

---

## Слайд 28. Критерий качества для TEST

Тестовый этап хороший, если:

- тесты проверяют модель, а не реализацию
- важные инварианты выражены в тестах
- есть unhappy paths
- есть destructive / update scenarios
- тесты падают (реализации ещё нет!)

**Красный флаг:**

Если тесты получились настолько слабыми, что под них можно подогнать почти любой reconcile-loop, значит контракт не работает.

---

## Слайд 29. CODE — реализация зафиксированного контракта

CODE — это не «пусть AI напишет решение». Это «пусть AI реализует уже зафиксированный контракт».

**Input:**

- spec
- implementation plan
- конкретная задача из PLAN
- **тесты, которые уже написаны и падают**
- project context

**Задача этапа:**

Написать код, который проходит тесты и выполняет POST conditions задачи.

**Выход этапа:**

- все тесты проходят
- `make build` и `go vet` проходят
- если обнаружены пробелы в spec — зафиксированы, а не решены молча

---

## Слайд 30. Context Engineering: что должен знать AI до генерации

Минимум контекста, который нужно держать в проекте:

- команды (`make generate`, `make test`, `go vet`, `golangci-lint`)
- known gotchas
- ссылка на spec

Это удобно хранить в `CLAUDE.md` или аналогичном проектном контексте.

**Главная мысль:**

Контекст — это не «вот тебе весь репозиторий».
Контекст — это **сжатая инженерная память проекта**.
Он не заменяет spec и plan. Он снижает потери между сессиями.

---

## Слайд 31. Пример содержимого `CLAUDE.md`

```text
# Commands
make generate
make test
go vet ./...
golangci-lint run

# Spec
See SPEC.md for full model, invariants, and edge cases

# Known Gotchas
Don't use r.Get() after r.Update() without re-fetch
meta.SetStatusCondition() needs observedGeneration
Deleting one namespace must not break full project reconciliation
Lowering project quota below current utilization is allowed; only new violations should be blocked
Webhook must be fast — read status, don't list pods
```

---

## Слайд 32. Практический prompt для CODE

```text
Реализуй задачу 7 из PLAN.md (Quota reconciliation).
Тесты уже написаны — код должен их проходить.
Если видишь пробел в spec — остановись и назови его, не решай сам.
```

---

## Слайд 33. Критерий качества для CODE

Этап выполнен хорошо, если:

- все тесты проходят
- POST conditions из PLAN выполнены
- `make build` и `go vet` проходят
- пробелы в spec зафиксированы, а не решены молча
- вы понимаете каждую строку, а не «вроде выглядит правдоподобно»

**Красный флаг:**

Если вы не можете объяснить, почему код написан именно так, а не иначе — вы потеряли контроль.

---

## Слайд 34. REVIEW — зачем нужен отдельный этап

**Input:**

- spec
- diff
- tests
- known project constraints

**Задача этапа:**

Найти semantic problems до merge.

**Выход:**

- список замечаний
- severity
- варианты исправления
- решение: accept / revise

**Главная мысль:**

AI хорошо генерирует.  
Но без review он так же быстро генерирует скрытые ошибки.

---

## Слайд 35. Reviewer и Implementer должны быть разведены

Рабочая схема:

- Terminal 1 / Session 1 — `Implementer`
- Terminal 2 / Session 2 — `Reviewer`

Reviewer видит diff свежим взглядом и не повторяет bias автора.

**Практический prompt:**

```text
Code review: git diff main...HEAD

For every issue:
- Describe the problem
- Explain why it matters
- Give 2-3 options (+ do nothing)
- For each: effort, risk, impact
- Give your recommendation and why
```

**Усиление:** если reviewer имеет доступ к базе знаний проекта (прошлые ревью, архитектурные решения, известные проблемы), качество ревью значительно выше. Это связывает этапы LEARN и REVIEW в цикл.

**Что сказать голосом:**

Такой формат заставляет модель не только критиковать, но и мыслить в терминах инженерных компромиссов.

---

## Слайд 36. Hooks и автоматические проверки

Что стоит включить сразу:

- форматирование
- линтер
- `go vet`
- unit/envtest
- запрет опасных команд
- stop-checklist перед завершением задачи

**Gotcha:** инструменты вроде gopls не успевают за скоростью генерации. Линтер и vet лучше запускать не на каждое сохранение, а на коммит — когда код уже стабилен.

**Важно:**

Hooks не заменяют review.
Они только автоматизируют нижний уровень контроля качества.

---

## Слайд 37. Критерий качества для REVIEW

Review хороший, если:

- он находит semantic bugs, а не только стиль
- привязан к spec
- проверяет correctness и safety
- приводит к небольшому числу понятных правок
- завершает шаг решением accept / revise

**Красный флаг:**

Если review заканчивается фразой «в целом выглядит нормально», это не review.

---

## Слайд 38. LEARN — почему это не факультативно

**Input:**

- то, что сломалось
- то, что сработало
- review findings
- новые gotchas
- финальные решения по модели

**Задача этапа:**

Сделать так, чтобы следующая сессия работала лучше предыдущей.

**Выход:**

- обновлённый `CLAUDE.md`
- known gotchas
- checklists
- project memory
- правила для следующих задач

**Главная мысль:**

Если знания не записаны явно, AI не стал умнее. Он просто помог в одной конкретной сессии.

---

## Слайд 39. Что именно фиксировать в LEARN

После каждого шага полезно сохранять:

- архитектурные решения
- неочевидные инварианты
- найденные баги и их причины
- project-specific conventions
- guardrails, которые реально помогли

**Практический prompt:**

```text
Что мы узнали из этой задачи?
Какие правила и gotchas нужно сохранить,
чтобы в следующей сессии не повторить те же ошибки?
Обнови CLAUDE.md секциями:
- Known Gotchas
- Review Checklist
```

---

## Слайд 40. Критерий качества для LEARN

Этап выполнен хорошо, если:

- следующую похожую задачу реально можно делать быстрее
- новый инженер сможет понять контекст проекта без устной передачи
- модель проекта стала более явной
- повторяющиеся ошибки превращены в правила

**Красный флаг:**

Если через неделю вы снова объясняете модели одни и те же базовые правила проекта — этап `LEARN` провален.

## Слайд 41. Вся цепочка как pipeline артефактов

**SPEC produces:**
- model
- invariants
- edge cases
- acceptance criteria

**PLAN consumes SPEC and produces:**
- tasks
- dependencies
- contracts

**TEST consumes SPEC + PLAN and produces:**
- executable contract (тесты падают — реализации ещё нет)

**CODE consumes PLAN + TEST and produces:**
- реализация, которая проходит тесты

**REVIEW consumes SPEC + TEST + DIFF and produces:**
- fixes / decisions

**LEARN consumes everything and produces:**
- reusable project memory

**Что сказать голосом:**

Вот в этом месте я бы хотел, чтобы вы перестали думать про LLM как про чат и начали думать про него как про участника pipeline.

## Слайд 42. Как это выглядело на Project Operator за 7 дней

### День 1–2
- `SPEC + PLAN`
- вопросы к модели
- фиксация инвариантов
- дожим edge cases

### День 3–5
- `TEST + CODE` по каждой задаче из PLAN
- тесты в чистой сессии → реализация в другой сессии
- reconcile / webhook / access / quota
- review после каждого шага

### День 6
- `REVIEW` финальный
- reviewer session по всему diff
- исправление semantic issues

### День 7
- `E2E + LEARN`
- final checks
- merge
- обновление project memory

## Слайд 43. На каких задачах этот подход работает лучше всего

Лучшие кандидаты:

- Kubernetes controllers and operators
- internal platform tooling
- API / CRD driven systems
- backend задачи с явной моделью и инвариантами
- refactoring при хорошем test harness

Менее удачные первые кандидаты:

- размытые product-фичи без чёткой модели
- большие cross-cutting refactors без границ
- задачи, где вы сами ещё не понимаете desired behavior
- «перепиши полрепозитория»

## Слайд 44. Анти-паттерны

- «Сделай всё сразу»
- «Вот тебе весь контекст мира»
- AI пишет и тесты, и реализацию без разделения
- один и тот же инстанс и пишет, и ревьюит
- большие diff на сотни строк
- review отсутствует
- known gotchas нигде не записаны
- после трёх провалов человек не вмешивается и продолжает долбить тот же prompt

**Короткая формула:**

Плохой процесс + AI = быстрый путь к плохому результату.

## Слайд 45. Что делать завтра в своей команде

1. Выделить одну реальную инженерную задачу с чёткой моделью
2. Перед кодом написать короткий spec
3. Разбить задачу на маленькие контракты с критерием готовности
4. Завести `CLAUDE.md` или аналог project context
5. Развести implementer / tester / reviewer роли
6. После каждой задачи обновлять project memory

## Слайд 46. Главный вывод

- LLM не заменяет инженерный процесс
- LLM усиливает инженерный процесс
- Чем сложнее система, тем важнее:
  - spec
  - декомпозиция
  - тестовый контракт
  - review
  - project memory
- Главная единица успеха — не prompt
- Главная единица успеха — **управляемый цикл разработки**

**Финальная фраза:**

**Не AI пишет код. Инженер пишет код руками AI.**
