# Как разрабатывать production Kubernetes-операторы с помощью LLM так, чтобы это можно было повторить у себя

Ниже — полная переписанная версия презентации под цель:

**не просто показать кейс, а дать слушателям воспроизводимый процесс**.

Главная ось всей презентации:

**SPEC → PLAN → TEST → CODE → REVIEW → LEARN**

Главная мысль:

**LLM ускоряет не сам по себе. Он ускоряет только тогда, когда встроен в инженерный процесс с явными артефактами, ограничениями и проверками.**

**Инженер, который сам пробует AI на сложных задачах.**
Вы попросили LLM сделать контроллер, получили 800 строк, задеплоили — а через час namespace'ы начали терять owner reference, потому что AI молча упростил модель владения.

Если вы уже тонули в пулл-реквестах от нейросетей и тратили три дня на дебаг кода, который был написан за минуту — этот доклад для вас. Я покажу процесс, который заставит ИИ работать на вас, а не против вас.

## Слайд 3. Главная проблема

- Почти все уже попробовали AI в разработке
- На простых задачах кажется, что всё ускорилось
- На сложных системах приходит хаос:
  - AI молча принимает архитектурные решения за вас
  - генерирует большие diff, которые невозможно ревьюить
  - упрощает модель там, где упрощать нельзя
  - создаёт ложное ощущение скорости — «код есть, контроля нет»

**Ключевой тезис:**

**Проблема не в том, что LLM плохо пишет код. Проблема в том, что большинство встраивает его в плохой процесс.**

**Что сказать голосом:**

Знакомое чувство? Вы получили pull request от AI, он компилируется, тесты проходят, но вы не можете объяснить, почему код устроен именно так. Это и есть потеря контроля.

## Слайд 5. Реальный кейс: Project Operator

Мы не делали игрушечный CRUD-controller.

Мы делали `Project Operator`, который:

- объединяет несколько namespace в один логический проект
- задаёт общие project-level quotas
- выдаёт доступы на уровне проекта
- разводит роли cluster admin и project admin
- синхронизирует desired state во всех namespace проекта

Основные ресурсы:

- `Project`
- `ProjectRole`
- `ProjectAccessBinding`

**Что сказать голосом:**

Это важный момент. Я специально показываю не trivial demo, а задачу, где есть модель владения, quota safety, access model, reconciliation, webhook validation и тестирование.

## Слайд 6. Почему именно оператор — хорошая проверка метода

Потому что здесь одновременно есть:

- доменная модель
- CRD schema design
- reconcile loop
- webhook validation
- cross-namespace ownership
- RBAC
- finalizers
- envtest / integration tests
- edge cases и partial failures

**Основная сложность:**

Не в том, чтобы написать 200 строк Go-кода.
А в том, чтобы **удержать модель целиком**.

**Что сказать голосом:**

Я взял оператор не потому, что это модно. Оператор — это идеальный стресс-тест. Здесь у вас доменная модель, идемпотентность, гонки, вебхуки и работа со стейтом кластера. Сложность не в том, чтобы написать двести строк на Go. Сложность в том, чтобы удержать всю эту конструкцию в голове и не уронить. Если фреймворк не ломается на k8s-операторе, он легко переварит любой ваш сложный бэкенд.

## Слайд 7. Что должен унести слушатель

После доклада человек должен уметь:

- понять, когда LLM вообще стоит использовать на backend/system design задаче
- начать не с генерации кода, а со спецификации
- разбить большую задачу на атомарные контракты
- заставить AI генерировать маленькие проверяемые куски
- отделить написание кода от написания тестов и review
- сохранять знания проекта так, чтобы следующая сессия была сильнее предыдущей

**Что сказать голосом:**

Этот метод не заменяет инженерный вкус. Он делает его переносимым. Без процесса архитектурные решения живут в голове одного сеньора и зависят от его памяти и удачи. С моим процессом — они зафиксированы в артефактах: SPEC.md, тестовый контракт, project memory. Вы унесете отсюда не вдохновение, а конвейер, который мы уже используем в проде.

## Слайд 8. Наивный подход, который ломает всё

Типичный запрос:

> Мне нужен Kubernetes-оператор, который объединяет namespace'ы в проекты, задаёт общие квоты и управляет доступом. Сделай.

Что происходит дальше:

- AI сам принимает архитектурные решения за вас
- сам придумывает quota model
- сам упрощает access model
- генерирует слишком большой diff
- смешивает бизнес-логику и побочные эффекты
- не удерживает edge cases
- пишет тесты под свою же реализацию
- создаёт опасную иллюзию прогресса

**Итог:**

Код есть, контроля нет.

**Что сказать голосом:**

Обычно знакомство с кодогенерацией выглядит так: мы просим LLM написать контроллер. Модель применяет лучшие паттерны... и молча удаляет нам все owner references, потому что ей так было проще. На игрушечных задачах можно позволить ИИ «просто набросать код». На production-операторе первый же правдоподобный кусок мусора становится фундаментом, который вы потом будете героически рефакторить.

## Слайд 9. Ключевой инсайт

**LLM — не автопилот. LLM — второй пилот.**

Разница между провалом и ускорением не в модели.  
Разница в процессе вокруг неё.

Плохой поток:

`idea → prompt → code → hope`

Хороший поток:

`SPEC → PLAN → TEST → CODE → REVIEW → LEARN`

**Что сказать голосом:**

Все вокруг твердят, что LLM — это ваш «второй пилот». Отличная метафора. Только наш второй пилот иногда решает, что крылья самолету в принципе не нужны — катиться по земле статистически безопаснее. Поэтому в моем фреймворке ИИ не принимает архитектурных решений. Решения принимаю я через спецификацию, а ИИ просто очень быстро итерирует внутри жестко заданных рамок.

Мой вклад — не набор промптов, а воспроизводимая рабочая модель: SPEC → PLAN → TEST → CODE → REVIEW → LEARN. У каждого этапа — вход, выход, критерий качества и отдельная роль для LLM.

На практике это меняет сам способ работы: тесты генерируются до кода в отдельной сессии, review отделяется от implementation, project memory фиксируется явно. Это не доклад про «как ещё использовать LLM». Это про то, как встроить его в production-grade engineering без потери контроля.

## Слайд 10. Главная схема доклада

**SPEC → PLAN → TEST → CODE → REVIEW → LEARN**

Это **рекомендуемая стартовая последовательность**.
В реальной работе — **итеративный цикл** с возвратами между этапами.

Для каждого этапа есть:

- вход
- действие
- выход
- критерий качества
- типичные ошибки

Каждый переход между этапами — точка принятия решения человеком. AI не двигается дальше без явного accept. Это human in the loop по дизайну, а не по случайности.

**Что сказать голосом:**

Зачем столько шагов для простой кодогенерации? Потому что ваше инженерное внимание стоит дорого. Писать шаблонный код — дешево. А вот разгребать последствия того, что ИИ тихо упростил модель владения — бесценно. Этот цикл нужен не для красоты. Он разгружает вашу память и не дает модели перескакивать через уровни абстракции.

LLM берёт на себя механическую часть внутри уже зафиксированного контракта. А человек остаётся в точках с высокой стоимостью решения: модель, границы, correctness, review. LLM повышает пропускную способность сильного инженера, если встроен в quality-controlled process.

Возвраты между этапами — норма, а не ошибка. Из CODE вы часто возвращаетесь в SPEC, когда обнаруживаете недосказанное. Из TEST — в PLAN, когда понимаете, что задача разбита неправильно. Но без этой опорной последовательности AI быстро размывает процесс.

## Слайд 11. Как читать остальные слайды

На каждом из следующих этапов я покажу 5 вещей:

1. Что является входом
2. Что делает человек
3. Чем помогает LLM
4. Какой артефакт должен появиться на выходе
5. Как понять, что шаг сделан достаточно хорошо

**Что сказать голосом:**

И сразу к вопросу, который может возникнуть: «Это реальная рабочая модель или красивая упаковка успешного кейса задним числом?» Это не схема, которой я объяснил успех. Это схема, через которую успех был произведён. Признак в том, что без артефактов предыдущего этапа следующий просто не запускался: без spec не было нормального plan, без test contract код быстро становился хаотичным, без review росла стоимость ошибок. И я прошёл этот цикл не один раз, а примерно на семи операторах за последние полгода. Поэтому для меня это operational framework, а не retrospective storytelling.

## Слайд 12. SPEC — зачем он нужен

**Input:**

- бизнес-задача
- ограничения
- существующая архитектура
- ваши реальные инварианты

**Задача этапа:**

Не дать AI принимать продуктовые и архитектурные решения за вас.

**Выход этапа:**

- spec-док
- список инвариантов
- список edge cases
- acceptance criteria
- список out-of-scope

**Главная мысль:**

Без spec AI решает за вас.
Со spec AI реализует ваши решения.

**Что сказать голосом:**

В классической парадигме мы часто додумываем архитектуру уже в процессе написания кода. С LLM это не работает. Первый же правдоподобный кусок сгенерированного кода становится фундаментом, который проще выкинуть, чем отрефакторить. Спека нужна, чтобы зафиксировать инварианты до того, как модель начнет фантазировать. Без спеки ИИ решает за вас. Со спекой — реализует ваши решения. Точка.

И да, spec уточняется по ходу — возвраты из CODE в SPEC у меня нормальная часть процесса. Но при spec first изменения происходят вокруг уже заданной модели, а при code first первый правдоподобный draft слишком быстро становится фактической архитектурой.

## Слайд 13. SPEC на примере Project Operator

В spec для `Project Operator` должны быть зафиксированы вещи вроде:

- `Project` объединяет несколько namespace в одну логическую сущность
- каждый namespace принадлежит ровно одному проекту
- project quota задаётся сверху и не должна нарушаться суммарно
- namespace attach происходит через label
- project admin живёт внутри ограничений, заданных cluster admin
- `ProjectAccessBinding` — основной user-facing способ выдачи доступа
- `ProjectRole` нужен только для кастомных ролей
- уменьшение project quota ниже текущей утилизации допустимо, но новые нарушения должны блокироваться отдельно

**Что сказать голосом:**

Посмотрите на этот список. Здесь нет ни строчки кода. Но именно это решает, ляжет ваш кластер или нет. Ни одна языковая модель в мире не скажет вам: «Слушай, а давай этот функционал сделаем подкомпонентом IAM, а не отдельным контроллером». Это знаете только вы и ваша архитектурная доска. ИИ напишет отличный код для абсолютно неправильной архитектуры, если вы не зафиксируете это в спеке.

**Реальный пример из нашего проекта:**

Когда ревьюер смотрел наш design doc, он заметил, что функциональность проектов должна быть частью компонента IAM, а не отдельным оператором. Мы мигрировали projects-operator как подкомпонент IAM. Это архитектурное решение, которое ни один AI не предложит — оно требует знания ландшафта вашей платформы.

## Слайд 14. Шаблон хорошего SPEC

Структура, которая работает:

- Цель
- Модель ресурсов
- Кто что может
- Инварианты
- Edge cases
- Failure modes
- Out of scope
- Acceptance criteria

**Пример инвариантов:**

- Every namespace MUST belong to exactly one `Project`
- Total project resource consumption MUST respect project limits
- Project admin MUST NOT be able to change project quota
- Reconciler MUST be idempotent

**Что сказать голосом:**

Мне не принципиален именно этот шаблон. Если у команды уже есть ADR, RFC или design note — отлично, используйте свой формат. Я продвигаю не шаблон, а необходимость явной и проверяемой формализации задачи. Чем хороший spec отличается от обычной Jira-тикетки или описания задачи? Минимум четыре вещи: что система должна делать, какие у неё инварианты и границы, какие edge cases и failure modes важны, и по каким признакам мы поймём, что задача решена правильно. Если документ не помогает независимо писать тесты и ревьюить решение — это не spec, а просто описание задачи.

## Слайд 15. Практический prompt для SPEC

```text
Мне нужен Kubernetes Project Operator.

Ресурсы:
- Project
- ProjectRole
- ProjectAccessBinding

Project объединяет namespace'ы в одну логическую сущность.
Namespace присоединяются к проекту через label.
У проекта есть общая квота CPU / memory / storage.
ProjectAccessBinding выдаёт доступ к проекту.
ProjectRole нужен только для кастомных ролей.
Cluster admin задаёт границы.
Project admin работает только внутри них.

Задай вопросы, которые нужны для написания полноценного spec.
Не предлагай код. Сначала только модель, инварианты, failure modes и edge cases.
```

**Что сказать голосом:**

Здесь ИИ работает как въедливый аналитик. Его задача — вытащить из вас edge cases до того, как они превратятся в баги. Если исходная постановка задачи слабая — без процесса вы всё равно напишете плохой код, просто сделаете это очень быстро. Мой процесс делает слабые места видимыми до того, как вы откроете IDE.

## Слайд 16. Devil’s Advocate после SPEC

После первого spec полезно добивать так:

```text
Выступи как критик этой модели.
Найди:
- race conditions
- ownership ambiguity
- quota safety issues
- partial failures
- delete/update edge cases
- authorization boundary mistakes

Сформулируй, каких решений в spec не хватает.
```

**Усиление:** для критичных архитектурных решений полезно получить независимые мнения от разных моделей ([llm-council](https://github.com/karpathy/llm-council)), а не просить одну модель критиковать саму себя.

**Выход:**

Обновлённый spec, в котором уже меньше сюрпризов на этапе кода.

**Что сказать голосом:**

Просить одну и ту же модель написать спеку и тут же её покритиковать — это как просить автора рецензировать собственную статью. Формально получите замечания, по факту — самоподтверждение. Для серьёзных архитектурных решений берите вторую модель или, ещё лучше, живого коллегу. Задача Devil's Advocate — найти дыры в вашей модели до того, как они станут багами в production.

## Слайд 17. Критерий качества для SPEC

SPEC считается достаточным, если:

- два инженера одинаково понимают desired behavior
- AI больше не придумывает модель сам
- по spec можно написать тесты до реализации
- основные edge cases уже зафиксированы

**Красный флаг:**

Если после чтения spec вы всё ещё не понимаете, что система должна делать при partial failure, delete, quota decrease или ownership conflict — spec ещё не готов.

**Что сказать голосом:**

Для меня хороший spec — это не красивый текст, а такой артефакт, по которому уже можно независимо написать тесты и два инженера одинаково понимают expected behavior. Вот это и есть критерий: spec поддерживает независимый дизайн тестов.

## Слайд 18. PLAN — зачем он нужен

**Input:**

- утверждённый spec

**Задача этапа:**

Разбить систему не на «большой оператор», а на последовательность маленьких инженерных задач.

**Выход этапа:**

- implementation plan
- очередность задач
- зависимости
- список файлов / пакетов / компонентов
- критерии готовности каждого шага

**Главная мысль:**

Большая задача почти всегда ломает LLM.
Маленькая задача с явным контрактом — почти всегда помогает.

**Что сказать голосом:**

План в голове разработчика — отличная штука. Жаль только, что его нельзя заревьюить и покрыть тестами. Пока план не записан, вы не можете ограничить ИИ. Большая задача с размытыми границами гарантированно ломает контекст модели. Мы разбиваем работу на атомы не ради бюрократии, а чтобы модель не перескакивала через уровни логики и не додумывала за нас бизнес-требования.

---

## Слайд 19. Задача + критерий готовности

Каждая подзадача формулируется так:

**Что сделать → Как понять, что сделано**

Примеры:

- Определить CRD types → `make generate` проходит без ошибок
- Namespace attachment → namespace'ы с label проекта появляются в `status.namespaces`
- Quota enforcement → webhook отклоняет pod, если суммарное потребление превышает `project.spec.quotas`
- Access reconciliation → RoleBinding существует в каждом namespace проекта для каждого ProjectAccessBinding
- Namespace validation webhook → namespace с label несуществующего проекта отклоняется

**Что сказать голосом:**

Критерий готовности — это главное. Без него задача размыта, и LLM будет додумывать за вас. С ним — у вас есть конкретная проверка, прошёл шаг или нет.

## Слайд 20. PLAN на примере Project Operator

Разумная последовательность шагов:

1. Scaffold проекта (kubebuilder init, cert-manager в kind)
2. Определить CRD types
3. Прогнать `make generate`
4–9. Для каждой задачи (namespace attachment, quota, access, webhooks, status):
   - сначала тесты (чистая сессия)
   - потом реализация (другая сессия)
10. Проверить покрытие, добить интеграционные тесты
11. Прогнать review и cleanup
12. Прогнать e2e / kind

**Что сказать голосом:**

Обратите внимание: тесты пишутся не в конце одним блоком, а перед каждой задачей. И обязательно в отдельной сессии — иначе AI пишет тесты под свой же код, а не под spec.

## Слайд 21. Pre/Post Conditions как контракт

Для каждой подзадачи полезно задавать:

**PRE:**

- какие сущности уже существуют
- какие assumptions допустимы
- что должно быть подготовлено заранее

**POST:**

- что станет истинно после выполнения шага
- что можно проверять тестом

**Пример:**

PRE:
- `Project` существует
- namespace'ы разрешимы
- CRD types уже сгенерированы

POST:
- все namespace с label проекта прикреплены к проекту
- суммарные quotas не превышают project limits
- access bindings отражают desired state

**Что сказать голосом:**

Без явных PRE и POST условий ваш план — это просто список желаний. Контракт — это жесткая граница: что должно быть на входе, и как изменится стейт на выходе. Когда у вас есть такие контракты, вы можете спокойно отдавать модели кусок логики и точно знать, что она не сломает соседний неймспейс просто потому, что у нее сбился контекст.

## Слайд 22. Практический prompt для PLAN

```text
У меня есть утверждённый spec для Project Operator.
Разбей реализацию на маленькие задачи.

Для каждой задачи укажи:
- имя
- цель
- зависимости
- PRE conditions
- POST conditions
- как проверить завершённость

Ограничения:
- одна задача = один логический шаг
- не смешивай CRD, reconciler, webhook и tests в одном большом шаге
```

**Что сказать голосом:**

Я не отдаю нейросети планирование архитектуры. Архитектурные границы я задал на шаге SPEC. Здесь ИИ работает просто как быстрый аналитик: он нарезает рутину внутри моих жестких заборов. Моя задача — проверить, не смешала ли модель в одном шаге вебхук, тесты и генерацию CRD. Если смешала — план отправляется в помойку.

## Слайд 23. Критерий качества для PLAN

PLAN считается хорошим, если:

- каждую задачу можно дать AI отдельно
- каждую задачу можно проверить независимо
- зависимости понятны заранее
- нет больших зон «сейчас просто сделаем всё остальное»

**Красный флаг:**

Если ваш план заканчивается пунктом «ну а дальше реализовать контроллер», это не план, а иллюзия плана.

**Что сказать голосом:**

Как понять, что plan хороший, до начала реализации? Не по красоте, а по проверяемости. Хороший plan — это не список шагов, а набор проверяемых задач с явными зависимостями. Конкретно: по каждой задаче можно отдельно сформулировать тестовый контракт, POST можно проверить независимо, и по задаче можно ожидать маленький reviewable diff. Негативные сигналы тоже видны до кода: задача слишком большая, смешивает CRD, reconciler, webhook и тесты одновременно, следующая задача не понимает, на что опирается. Если из задачи нельзя сделать отдельный тестовый контракт — план ещё не готов, даже если выглядит аккуратно.

## Слайд 24. TEST — почему тесты идут до кода

**Input:**

- spec
- implementation plan
- конкретная задача из PLAN
- acceptance criteria

**Задача этапа:**

Превратить требования и инварианты в проверяемый контракт **до** написания кода.

**Выход:**

- unit / envtest / integration tests
- тесты компилируются, но падают (реализации ещё нет)
- по тестам можно понять expected behavior системы

**Главная мысль:**

Если код уже в контексте, AI пишет тесты под реализацию, включая её баги. Тесты пишутся в чистой сессии — без контекста кода.

**Что сказать голосом:**

TEST перед CODE — это не TDD-ритуал и не сценический трюк. Дело в том, что одного spec недостаточно. Spec остаётся текстом, а тесты превращают expected behavior в исполняемый контракт.

Когда код уже попал в контекст модели, она начинает мыслить от реализации и легко подгоняет тесты под уже выбранное решение. Если тесты появляются раньше и человек их отдельно ревьюит, то у реализации появляется жёсткая внешняя рамка — separate reviewed test boundary.

Для Kubernetes-операторов это особенно важно, потому что там много скрытых failure modes: idempotency, partial failures, delete path, webhook validation, quota safety. Текстом про них легко сказать «учтено», а тестами — нужно реально зафиксировать ожидаемое поведение.

## Слайд 25. Правильный поток TDD с AI

Неправильно:

- «Напиши код и сразу тесты к нему»

Правильно:

1. В чистой сессии попросить тесты по SPEC и PLAN
2. Человеку прочитать тесты
3. Закоммитить тесты как контракт
4. В другой сессии просить реализацию, которая проходит эти тесты

Формула:

**tests first, implementation second, разные сессии**

**Практический нюанс:**

На практике порядок может быть и TEST→CODE, и CODE→TEST. Главное правило — **разные сессии**. Тот, кто пишет тесты, не должен видеть реализацию в контексте.

- TEST→CODE — безопаснее для критичных мест (webhooks, quota, access control)
- CODE→TEST — быстрее, когда спек чёткий и задача понятна

**Что сказать голосом:**

Писать тесты до кода с ИИ — это не TDD-ритуал для галочки. Это бетонный забор. Если показать модели реализацию до написания тестов, она напишет идеальные тесты, которые заботливо покроют и обоснуют все её же галлюцинации. Тесты пишутся в чистой сессии. И только потом, в другой сессии, мы просим модель в этот контракт вписаться.

Spec задаёт рамку словами. Tests задают рамку исполняемо. Поэтому TEST before CODE — это способ превратить вариативный output модели в управляемую реализацию внутри явного контракта.

## Слайд 26. Практический prompt для TEST

```text
Напиши envtest тесты для task-07 из PLAN.md.

Правила:
- сценарии брать только из acceptance criteria в SPEC.md
- писать только тесты
- reconciler не реализовывать
- тесты должны выражать expected behavior, а не детали реализации
- тесты должны компилироваться, но падать

Выведи:
1) список сценариев
2) сами тесты
3) какие части контракта пока не покрыты
```

**Что сказать голосом:**

Доверяю ли я модели писать тесты? Безусловно, нет. Она просто пишет первый черновик исполняемого контракта, опираясь строго на спеку. Моя работа здесь — проверить три вещи: выражают ли тесты реальное поведение, закрыты ли корнер-кейсы и не прибиты ли они гвоздями к какой-то конкретной реализации, которой еще даже не существует.

Минимальный набор:

- happy path
- idempotency
- partial failures
- update path
- delete path
- authorization boundaries
- admission validation
- edge cases бизнес-модели

## Слайд 28. Критерий качества для TEST

Тестовый этап хороший, если:

- тесты проверяют модель, а не реализацию
- важные инварианты выражены в тестах
- есть unhappy paths
- есть destructive / update scenarios
- тесты падают (реализации ещё нет!)

**Красный флаг:**

Если тесты получились настолько слабыми, что под них можно подогнать почти любой reconcile-loop, значит контракт не работает.

**Что сказать голосом:**

Главное правило: хороший тест можно прочитать и понять, как работает система, вообще не заглядывая в код реализации. Если тесты получились такими водянистыми, что под них можно подогнать любой кривой reconcile-loop — значит, ваш контракт дырявый. Идите и переписывайте тесты, пока к коду даже не притрагивались.

## Слайд 29. CODE — реализация зафиксированного контракта

CODE — это не «пусть AI напишет решение». Это «пусть AI реализует уже зафиксированный контракт».

**Input:**

- spec
- implementation plan
- конкретная задача из PLAN
- **тесты, которые уже написаны и падают**
- project context

**Задача этапа:**

Написать код, который проходит тесты и выполняет POST conditions задачи.

**Выход этапа:**

- все тесты проходят
- `make build` и `go vet` проходят
- если обнаружены пробелы в spec — зафиксированы, а не решены молча

**Что сказать голосом:**

Да, на этом этапе ИИ — это просто очень быстрый template expander. И в этом нет проблемы. Инженерное открытие никуда не исчезло, оно просто сместилось влево: в архитектуру, декомпозицию и контракты. Скорость печати символов дает модель, а управляемость этой скорости даете вы.

## Слайд 30. Context Engineering: что должен знать AI до генерации

Минимум контекста, который нужно держать в проекте:

- команды (`make generate`, `make test`, `go vet`, `golangci-lint`)
- known gotchas
- ссылка на spec

Это удобно хранить в `CLAUDE.md` или аналогичном проектном контексте.

**Главная мысль:**

Контекст — это не «вот тебе весь репозиторий».
Контекст — это **сжатая инженерная память проекта**.
Он не заменяет spec и plan. Он снижает потери между сессиями.

**Что сказать голосом:**

Мы могли бы скормить модели весь репозиторий, чтобы она окончательно сошла с ума в поисках нужных абстракций. Вместо этого мы даем ей `CLAUDE.md`. Контекст — это не дамп исходников. Это выжимка боли и крови вашего проекта: жесткие конвенции, команды сборки и список граблей, на которые не надо наступать дважды.

Как отличить проблему контекста от проблемы модели? Если модель придумывает свою архитектуру, дублирует существующие helper-ы, игнорирует стиль — ей не хватает контекста. Если контекст достаточный, а логика всё равно слабая — это ограничение reasoning самой модели.

---

## Слайд 31. Пример содержимого `CLAUDE.md`

```text
# Commands
make generate
make test
go vet ./...
golangci-lint run

# Spec
See SPEC.md for full model, invariants, and edge cases

# Known Gotchas
Don't use r.Get() after r.Update() without re-fetch
meta.SetStatusCondition() needs observedGeneration
Deleting one namespace must not break full project reconciliation
Lowering project quota below current utilization is allowed; only new violations should be blocked
Webhook must be fast — read status, don't list pods
```

**Что сказать голосом:**

Мне скажут: «Зачем отдельный файл? Эти знания могут жить в comments, tests, ADR, README.» Да, и они должны там жить. CLAUDE.md — это не ещё один источник истины, а короткий operating brief поверх уже существующих источников истины. Его задача узкая: как жить в этом проекте, где искать детали, какие локальные invariants важны и какие ошибки нельзя повторять. Подробности живут в docs, ADR, tests и comments, а CLAUDE.md только ссылается на них и фиксирует durable gotchas и conventions. И он обновляется не после каждой задачи, а только когда появляется знание, которое реально изменит следующий цикл работы. Если знание должно жить подробно — ему не место в CLAUDE.md. Там должна жить только его короткая operational выжимка или ссылка.

## Слайд 32. Практический prompt для CODE

```text
Реализуй task-07 из PLAN.md: quota reconciliation.

Контекст:
- SPEC.md зафиксирован
- тесты уже написаны и падают
- acceptance criteria менять нельзя

Правила:
- сначала коротко опиши план реализации (5–10 строк)
- реализация должна пройти существующие тесты
- если видишь пробел в spec — остановись и сформулируй вопрос
- не меняй тесты, если это не явная ошибка в контракте
- держи diff маленьким и reviewable (<200 строк)

Выведи: план, затем код
```

**Что сказать голосом:**

Обратите внимание, в промпте нет никаких магических слов в стиле «будь сеньор-разработчиком с 20-летним стажем». Есть жесткая ссылка на спеку, падающие тесты и лимит по строкам. Промпт — это просто интерфейс к вашим инженерным артефактам. ИИ здесь выполняет роль очень быстрого принтера, который печатает код внутри железобетонного контура.

## Слайд 33. Критерий качества для CODE

Этап выполнен хорошо, если:

- все тесты проходят
- POST conditions из PLAN выполнены
- `make build` и `go vet` проходят
- пробелы в spec зафиксированы, а не решены молча
- вы понимаете каждую строку, а не «вроде выглядит правдоподобно»

**Красный флаг:**

Если вы не можете объяснить, почему код написан именно так, а не иначе — вы потеряли контроль.

**Что сказать голосом:**

Зеленые тесты — это еще не повод пушить в мейн. Код считается хорошим, только если вы можете объяснить, почему каждая строчка написана именно так. Если вы смотрите на генерацию и думаете «ну, вроде выглядит правдоподобно» — поздравляю, вы только что потеряли контроль над собственным проектом.

Production-свойства — идемпотентность, delete path, partial failures — должны быть зафиксированы раньше в pipeline: в SPEC как invariants, в TEST как сценарии, в REVIEW как checklist. CODE их реализует, а не изобретает.

## Слайд 34. REVIEW — зачем нужен отдельный этап

**Input:**

- spec
- diff
- tests
- known project constraints

**Задача этапа:**

Найти semantic problems до merge.

**Выход:**

- список замечаний
- severity
- варианты исправления
- решение: accept / revise

**Главная мысль:**

AI хорошо генерирует.
Но без review он так же быстро генерирует скрытые ошибки.

**Что сказать голосом:**

Мне скажут: «Если SPEC сильный и TEST правильные, зачем ещё REVIEW?» Потому что тесты проверяют, что код ведёт себя правильно. Review проверяет, что он устроен разумно. Это разные вещи. Review ловит не «код не работает», а «код работает, но выбран плохой или опасный способ»: лишняя сложность, скрытые риски по scope доступа, хрупкое решение при формально зелёных тестах, код, который непонятен следующему инженеру. Review нужен, чтобы поймать плохое решение, которое всё ещё проходит хороший контракт. И если по коду видно, что tests или spec были недостаточными — review тоже должен это поднять. Он не дублирует их роль, но может обнаружить их слабость.

## Слайд 35. Reviewer и Implementer должны быть разведены

Рабочая схема:

- Terminal 1 / Session 1 — `Implementer`
- Terminal 2 / Session 2 — `Reviewer`

Reviewer видит diff свежим взглядом и не повторяет bias автора.

**Как реализовать:** `git worktree` — физическое разделение контекста. Reviewer работает в отдельной копии репо и не видит незакоммиченный код автора. Это же работает для разделения TEST и CODE сессий.

**Практический prompt:**

```text
Code review: git diff main...HEAD

For every issue:
- Describe the problem
- Explain why it matters
- Give 2-3 options (+ do nothing)
- For each: effort, risk, impact
- Give your recommendation and why
```

**Усиление:** если reviewer имеет доступ к базе знаний проекта (прошлые ревью, архитектурные решения, известные проблемы), качество ревью значительно выше. Это связывает этапы LEARN и REVIEW в цикл.

**Автоматический review на PR:**

Copilot / CodeRabbit на каждый PR дают ещё один слой ревью. В нашем случае автоматический ревью нашёл реальные баги: cluster-scoped CRD помечены как namespaced в PROJECT, неправильный тип printer column, пропущенная валидация. Из ~15 комментариев 4–5 оказались полезными — остальное шум по scaffold-коду.

Пример: [PR #1](https://github.com/azalio/devopsconf-ai-develop/pull/1) — ревью Copilot на scaffold + CRD types.

**Реальные примеры из review нашего оператора:**

Ревьюер нашёл три класса проблем, которые AI создал молча:

1. **Лишняя абстракция.** AI обернул `rules` в `ProjectRoleSpec`, хотя нативные `Role`/`ClusterRole` держат rules на верхнем уровне. AI следовал шаблону kubebuilder, а не convention Kubernetes API. Ревьюер попросил убрать Spec-обёртку — сделали `rules` top-level, как в нативных ресурсах.

2. **Генерация «на вырост».** AI создал 4 фазы `ProjectNamespacePhase` (Pending, Active, Terminating, Failed), но в коде реально использовалась только Active. Формально всё правильно, но это мусор, который усложняет API без причины.

3. **Расхождение кода и документации.** Комментарий к `AllowDeletion` описывал удаление namespace, а реализация блокировала удаление ресурса `ProjectNamespace` через webhook. Тонкая, но опасная разница.

Ни одну из этих проблем нельзя поймать линтером или тестом. Только человек с пониманием модели.

**Что сказать голосом:**

Генерировать код и ревьюить его в одном окне чата — это как просить студента самому себе поставить оценку за диплом. У нас это физически разные сессии. Модель-ревьюер не независима, но это дешевый, безжалостный второй проход по контракту. Она ловит сломанный RBAC и разъехавшуюся бизнес-логику до того, как этот мусор попадет ко мне на PR.

Вот эти три примера на слайде — чисто семантические, и без живого ревьюера они бы уехали в прод. Reviewer-LLM не независима как человек, но полезна как дешёвый второй проход по явному контракту. Это снимает с инженера рутинную проверку и оставляет ему больше внимания на semantic correctness и архитектурные компромиссы.

## Слайд 36. Hooks и автоматические проверки

Что стоит включить сразу:

- форматирование
- линтер
- `go vet`
- unit/envtest
- запрет опасных команд
- stop-checklist перед завершением задачи

**Gotcha:** инструменты вроде gopls не успевают за скоростью генерации. Линтер и vet лучше запускать не на каждое сохранение, а на коммит — когда код уже стабилен.

**Важно:**

Hooks не заменяют review.
Они только автоматизируют нижний уровень контроля качества.

**Что сказать голосом:**

Hooks не делают код правильным. Они делают ревью менее шумным. Автоматизировать нужно всё, что проверяется дешево: линтеры, форматирование, компиляцию. Но hooks принципиально не отвечают на вопросы бизнес-логики. Всё, что требует инженерного суждения, остается на человеке.

## Слайд 37. Критерий качества для REVIEW

Review хороший, если:

- он находит semantic bugs, а не только стиль
- привязан к spec
- проверяет correctness и safety
- приводит к небольшому числу понятных правок
- завершает шаг решением accept / revise

**Красный флаг:**

Если review заканчивается фразой «в целом выглядит нормально», это не review.

**Что сказать голосом:**

Если ревьюер — живой или искусственный — пишет вам «в целом выглядит нормально», это не ревью. Это отписка. Хорошее ревью всегда пытается сломать вашу логику об исходную спеку и тесты. Признак качественного ревью не в количестве докопок до нейминга переменных, а в том, нашли ли вы семантические дыры в логике до того, как они уехали в прод.

## Слайд 38. LEARN — почему это не факультативно

**Input:**

- то, что сломалось
- то, что сработало
- review findings
- новые gotchas
- финальные решения по модели

**Задача этапа:**

Сделать так, чтобы следующая сессия работала лучше предыдущей.

**Выход:**

- обновлённый `CLAUDE.md`
- known gotchas
- checklists
- project memory
- правила для следующих задач

**Главная мысль:**

Если знания не записаны явно, AI не стал умнее. Он просто помог в одной конкретной сессии.

**Что сказать голосом:**

Частый вопрос: «А CLAUDE.md не превратится в свалку?» Ответ — нет, если относиться к нему как к коду, а не как к блокноту. В нашем случае CLAUDE.md лежит в репозитории, версионируется и проходит review через PR — точно так же, как любой другой файл проекта. Это управляемый артефакт, а не личный дневник.

## Слайд 39. Что именно фиксировать в LEARN

После каждого шага полезно сохранять:

- архитектурные решения
- неочевидные инварианты
- найденные баги и их причины
- project-specific conventions
- guardrails, которые реально помогли

**Практический prompt:**

```text
Что мы узнали из этой задачи?
Какие правила и gotchas нужно сохранить,
чтобы в следующей сессии не повторить те же ошибки?
Обнови CLAUDE.md секциями:
- Known Gotchas
- Review Checklist
```

**Что сказать голосом:**

Обратите внимание: в CLAUDE.md попадает не всё, что мы узнали, а только то, что доказало ценность в принятом инженерном цикле. Четыре категории: команды сборки и запуска, инварианты модели, known gotchas и review checklist. Каждое правило должно быть заработано — оно появляется, потому что без него реально ломалось. И обратный процесс тоже работает: если правило перестало быть актуальным — оно удаляется на review, как deprecated код. Поэтому CLAUDE.md не растёт бесконечно — он живёт в том же цикле, что и остальной проект.

## Слайд 40. Критерий качества для LEARN

Этап выполнен хорошо, если:

- следующую похожую задачу реально можно делать быстрее
- новый инженер сможет понять контекст проекта без устной передачи
- модель проекта стала более явной
- повторяющиеся ошибки превращены в правила

**Красный флаг:**

Если через неделю вы снова объясняете модели одни и те же базовые правила проекта — этап `LEARN` провален.

**Что сказать голосом:**

Если через неделю вы снова объясняете модели в промпте, что в вашем проекте нельзя трогать чужие лейблы — вы с треском провалили этап LEARN. Наша цель — сделать так, чтобы каждая следующая задача в этом репозитории делалась быстрее и чище. Мы фиксируем ошибки в `CLAUDE.md`, и проект буквально учится не повторять свои же косяки.

## Слайд 41. Вся цепочка как pipeline артефактов

**SPEC produces:**
- model
- invariants
- edge cases
- acceptance criteria

**PLAN consumes SPEC and produces:**
- tasks
- dependencies
- contracts

**TEST consumes SPEC + PLAN and produces:**
- executable contract (тесты падают — реализации ещё нет)

**CODE consumes PLAN + TEST and produces:**
- реализация, которая проходит тесты

**REVIEW consumes SPEC + TEST + DIFF and produces:**
- fixes / decisions

**LEARN consumes everything and produces:**
- reusable project memory

**Что сказать голосом:**

Вот в этом месте я бы хотел, чтобы вы перестали думать про LLM как про чат и начали думать про него как про участника pipeline.

И тут резонный вопрос: «А не создаёт ли это иллюзию контроля? Много этапов, артефактов, quality gates — но ошибки-то всё равно возможны.» Да, любой процесс может создать ложное ощущение контроля. Но посмотрите на эту схему: каждый этап ловит свой класс ошибок. Дыры в модели всплывают на SPEC. Невозможность выразить expected behavior — на TEST. Расползание scope — на CODE, потому что diff маленький. Semantic bugs — на REVIEW, потому что reviewer проверяет против spec и тестов. Я не убираю риск ошибки. Я двигаю обнаружение ошибки как можно левее по pipeline. А хаотичная работа с LLM создаёт ту же иллюзию контроля — просто без точек, где её можно остановить.

## Слайд 43. На каких задачах этот подход работает лучше всего

Лучшие кандидаты:

- Kubernetes controllers and operators
- internal platform tooling
- API / CRD driven systems
- backend задачи с явной моделью и инвариантами

Этот процесс окупается там, где когнитивная нагрузка и так уже пробивает потолок: новые доменные модели, хитрый стейт-менеджмент, квоты и доступы. Мы не добавляем оверхед, мы берем уже существующий хаос сложной задачи и делаем его управляемым.

## Слайд 44. Анти-паттерны

- «Сделай всё сразу»
- «Вот тебе весь контекст мира»
- AI пишет и тесты, и реализацию без разделения
- один и тот же инстанс и пишет, и ревьюит
- большие diff на сотни строк
- review отсутствует
- known gotchas нигде не записаны
- после трёх провалов человек не вмешивается и продолжает долбить тот же prompt

**Реальный пример:**

AI сгенерировал `ProjectRole` со стандартной kubebuilder-структурой `Spec`/`Status`, хотя по API convention Kubernetes `rules` должны быть top-level. Выглядит профессионально, компилируется, тесты проходят — но это неправильный API design. Без ревьюера, который знает нативные ресурсы, это бы уехало в прод.

**Короткая формула:**

Плохой процесс + AI = быстрый путь к плохому результату.

**Что сказать голосом:**

Плохой инженерный процесс плюс быстрый ИИ равно катастрофа в проде на огромной скорости. AI генерирует `ProjectRole` с левыми структурами, всё компилируется, тесты зеленые. Без человека, который понимает нативные API-конвенции Kubernetes, этот мусор просочится в релиз. LLM не избавляет вас от необходимости фильтровать бред, она этот бред масштабирует.

Граница простая: как только output дешевле сгенерировать, чем качественно отревьюить — LLM перестаёт помогать. Мой процесс — small diffs, separate review, tests before code — как раз нужен, чтобы не переходить эту границу.

## Слайд 45. Что делать завтра в своей команде

1. Выделить одну реальную инженерную задачу с чёткой моделью
2. Перед кодом написать короткий spec
3. Разбить задачу на маленькие контракты с критерием готовности
4. Завести `CLAUDE.md` или аналог project context
5. Развести implementer / tester / reviewer роли
6. После каждой задачи обновлять project memory

**Что сказать голосом:**

И тут важный вопрос: «А как это внедрять в команде из пяти-семи человек, чтобы процесс не превратился в бюрократический театр?» Ответ: не внедрять как полный фреймворк на все задачи. Командное внедрение для меня — это минимальный набор артефактов и ротация ролей на сложных задачах. Первое: выбрать один класс задач, где процесс реально нужен — контроллеры, операторная логика, сложные backend-фичи с инвариантами. Не тащить это на мелкие багфиксы. Второе: минимальные артефакты — короткий spec, plan с маленькими задачами, тестовый контракт до реализации, отдельный review. Третье: лёгкая ротация ролей — один пишет spec, другой его читает, третий ревьюит план или тестовый контракт. Автор кода не является единственным reviewer своего diff. Тогда процесс включается только там, где есть реальная сложность, и помогает команде, а не начинает жить своей жизнью.

## Слайд 46. Главный вывод

- LLM не заменяет инженерный процесс
- LLM усиливает инженерный процесс
- Чем сложнее система, тем важнее:
  - spec
  - декомпозиция
  - тестовый контракт
  - review
  - project memory
- Главная единица успеха — не prompt
- Главная единица успеха — **управляемый цикл разработки**

**Финальная фраза:**

**Не AI пишет код. Инженер пишет код руками AI.**

**Что сказать голосом:**

Цена рутины упала до нуля, а цена архитектурной ошибки взлетела в космос. Фреймворк, который я показал — это способ управлять этой новой сложностью. Инженер не стал придатком к промпту. Наоборот. Теперь вы можете удерживать архитектуру целого куска платформы в одиночку, потому что процесс делает вашу экспертизу масштабируемой. Не AI пишет код. Инженер пишет код руками AI.

И если спросят: «А если модели станут в пять раз лучше?» — тем более нужен процесс. Чем мощнее инструмент, тем дороже обходится хаотичная работа с ним. Мой доклад не про workaround под слабые LLM, а про безопасное встраивание сильных LLM в инженерную работу.

## Слайд 47. Чек-лист: один слайд, который стоит сфотографировать

```
SPEC:  ✓ Инварианты записаны
       ✓ Edge cases зафиксированы
       ✓ Out-of-scope явно назван
       ✓ Два инженера одинаково понимают desired behavior

PLAN:  ✓ Каждая задача имеет PRE/POST conditions
       ✓ Критерий готовности для каждого шага
       ✓ Нет задач «ну и дальше реализовать остальное»

TEST:  ✓ Тесты в отдельной сессии, без кода в контексте
       ✓ Тесты проверяют модель, а не реализацию
       ✓ Есть unhappy paths и destructive scenarios

CODE:  ✓ Пробелы в spec фиксируются, а не решаются молча
       ✓ make build + go vet проходят
       ✓ Вы понимаете каждую строку

REVIEW:✓ Reviewer ≠ implementer
       ✓ Проверка по spec, а не «выглядит нормально»
       ✓ Semantic bugs, а не только стиль

LEARN: ✓ CLAUDE.md / project memory обновлён
       ✓ Gotchas записаны
       ✓ Следующая сессия сильнее предыдущей
```

**Что сказать голосом:**

Этот чек-лист полезен не только когда процесс работает, но и когда он ломается. Если команда попробовала и не получилось — я бы не говорил «значит плохо делали». Я бы смотрел на артефакты. Spec расплывчатый и без acceptance criteria — проблема в постановке. Plan не даёт маленьких проверяемых задач — проблема в декомпозиции. Тесты слабые или написаны после кода — проблема в контракте. Diff большой и review формальный — проблема в process discipline. И только если всё это в порядке, а output всё равно слабый — значит либо модель не подходит, либо задача плохо ложится на такой способ работы. Сильный процесс полезен не только когда он работает, но и когда он ломается — потому что тогда видно, где именно он сломался. Без процесса команда просто говорит «LLM какая-то плохая». С процессом можно диагностировать конкретный сломанный этап.
