## Слайд 5. Главная проблема

- Почти все уже попробовали AI в разработке
- На простых задачах кажется, что всё ускорилось
- На сложных системах приходит хаос:
  - AI молча принимает архитектурные решения за вас
  - генерирует большие diff, которые невозможно ревьюить
  - упрощает модель там, где упрощать нельзя
  - создаёт ложное ощущение скорости — «код есть, контроля нет»

**Ключевой тезис:**

**Проблема не в том, что LLM плохо пишет код. Проблема в том, что большинство встраивает его в плохой процесс.**

**Что сказать голосом:**

Знакомое чувство? Вы получили pull request от AI, он компилируется, тесты проходят, но вы не можете объяснить, почему код устроен именно так. Это и есть потеря контроля.

## Слайд 6. Реальный кейс: Project Operator

Мы не делали игрушечный CRUD-controller.

Мы делали `Project Operator`, который:

- объединяет несколько namespace в один логический проект
- задаёт общие project-level quotas с real-time enforcement
- выдаёт доступы на уровне проекта
- разводит роли cluster admin и project admin
- синхронизирует desired state во всех namespace проекта

**Масштаб задачи:**

| | |
|---|---|
| **4 CRD** | Project, ProjectNamespace, ProjectRole, ProjectRoleBinding |
| **6 reconcilers** | включая отдельный QuotaReconciler и NamespaceWatcher |
| **7 webhooks** | валидация, мутация, RBAC-защита, квотный enforcement |
| **~22K строк Go** | production-код |
| **~29K строк тестов** | тестов больше, чем кода |

**Что сказать голосом:**

Посмотрите на эти цифры. 4 CRD, 6 reconcilers, 7 webhooks. Одна ProjectRole разворачивается в N Role-объектов во все namespace проекта. Квотная система работает в двух режимах: real-time admission webhook блокирует создание подов при превышении, а периодический reconciler пересчитывает утилизацию каждые 30 секунд. Это не toy project — это оператор, на котором стоит мультитенантность Stackland.

## Слайд 7. Почему именно оператор — хороший стресс-тест

Потому что здесь одновременно есть: доменная модель, CRD schema design, reconcile loop, webhook validation, cross-namespace ownership, RBAC, finalizers, envtest / integration tests, edge cases и partial failures.

**Основная сложность:**

Не в том, чтобы написать 200 строк Go-кода.
А в том, чтобы **удержать модель целиком**.

**Что сказать голосом:**

Я взял оператор не потому, что это модно. Оператор — это идеальный стресс-тест. Здесь у вас доменная модель, идемпотентность, гонки, вебхуки и работа со стейтом кластера. Сложность не в том, чтобы написать двести строк на Go. Сложность в том, чтобы удержать всю эту конструкцию в голове и не уронить. Если фреймворк не ломается на k8s-операторе, он легко переварит любой ваш сложный бэкенд.

## Слайд 8. Наивный подход, который ломает всё

Типичный запрос:

> Мне нужен Kubernetes-оператор, который объединяет namespace'ы в проекты, задаёт общие квоты и управляет доступом. Сделай.

Что происходит: AI сам принимает архитектурные решения, сам придумывает quota model, сам упрощает access model, генерирует слишком большой diff, пишет тесты под свою же реализацию.

**Итог:** Код есть, контроля нет.

**Что сказать голосом:**

Обычно знакомство с кодогенерацией выглядит так: мы просим LLM написать контроллер. Модель применяет лучшие паттерны... и молча удаляет нам все owner references, потому что ей так было проще. На игрушечных задачах можно позволить ИИ «просто набросать код». На production-операторе первый правдоподобный, но неверный кусок кода становится фундаментом, который потом очень дорого переделывать.

## Слайд 9. Ключевой инсайт

**LLM — не автопилот. LLM — второй пилот.**

Разница между провалом и ускорением не в модели.
Разница в процессе вокруг неё.

Плохой поток: `idea → prompt → code → hope`

Хороший поток: `SPEC → PLAN → TEST → CODE → REVIEW → LEARN`

**Что сказать голосом:**

Все вокруг твердят, что LLM — это ваш «второй пилот». Отличная метафора. Только наш второй пилот иногда решает, что крылья самолету в принципе не нужны — катиться по земле статистически безопаснее. Поэтому в моем фреймворке ИИ не принимает архитектурных решений. Решения принимаю я через спецификацию, а ИИ просто очень быстро итерирует внутри жестко заданных рамок.

Мой вклад — не набор промптов, а воспроизводимая рабочая модель: SPEC → PLAN → TEST → CODE → REVIEW → LEARN. У каждого этапа — вход, выход, критерий качества и отдельная роль для LLM.

---

## Слайд 10. Главная схема доклада

**SPEC → PLAN → TEST → CODE → REVIEW → LEARN**

Это **рекомендуемая стартовая последовательность**.
В реальной работе — **итеративный цикл** с возвратами между этапами.

Каждый переход между этапами — точка принятия решения человеком. AI не двигается дальше без явного accept.

**Что сказать голосом:**

Зачем столько шагов для простой кодогенерации? Потому что ваше инженерное внимание стоит дорого. Писать шаблонный код — дешево. А вот разгребать последствия того, что ИИ тихо упростил модель владения — бесценно. Этот цикл нужен не для красоты. Он разгружает вашу память и не дает модели перескакивать через уровни абстракции.

Возвраты между этапами — норма, а не ошибка. Из CODE вы часто возвращаетесь в SPEC, когда обнаруживаете недосказанное. Из TEST — в PLAN, когда понимаете, что задача разбита неправильно. Но без этой опорной последовательности AI быстро размывает процесс.

## Слайд 11. SPEC: что фиксируем до кода

**Без spec AI решает за вас. Со spec AI реализует ваши решения.**

Инварианты из нашего Project Operator:

- `Project` объединяет несколько namespace в одну логическую сущность
- каждый namespace принадлежит ровно одному проекту
- project quota задаётся сверху и не должна нарушаться суммарно
- project admin живёт внутри ограничений, заданных cluster admin
- `ProjectAccessBinding` — основной user-facing способ выдачи доступа
- уменьшение project quota ниже текущей утилизации допустимо, но новые нарушения блокируются

**Структура хорошего spec:** цель, модель ресурсов, кто что может, инварианты, edge cases, failure modes, out of scope, acceptance criteria.

**Красный флаг:** Если по spec нельзя независимо написать тесты — spec не готов.

**Реальный пример:**

Когда ревьюер смотрел наш design doc, он заметил, что функциональность проектов должна быть частью компонента IAM, а не отдельным оператором. Мы мигрировали projects-operator как подкомпонент IAM. Это архитектурное решение, которое ни один AI не предложит — оно требует знания ландшафта вашей платформы.

**Что сказать голосом:**

В классической парадигме мы часто додумываем архитектуру уже в процессе написания кода. С LLM это не работает. Первый же правдоподобный кусок сгенерированного кода становится фундаментом, который проще выкинуть, чем отрефакторить. Спека нужна, чтобы зафиксировать инварианты до того, как модель начнет фантазировать.

Посмотрите на этот список. Здесь нет ни строчки кода. Но именно это решает, ляжет ваш кластер или нет. ИИ напишет отличный код для абсолютно неправильной архитектуры, если вы не зафиксируете это в спеке.

## Слайд 12. Как я мучаю модель на этапе SPEC

**Шаг 1 — извлечь требования:**

```text
Мне нужен Kubernetes Project Operator.
Project объединяет namespace'ы в одну логическую сущность.
У проекта есть общая квота CPU / memory / storage.
ProjectAccessBinding выдаёт доступ к проекту.

Задай вопросы, которые нужны для написания полноценного spec.
Не предлагай код. Сначала только модель, инварианты, failure modes и edge cases.
```

**Шаг 2 — Devil's Advocate:**

```text
Выступи как критик этой модели.
Найди: race conditions, ownership ambiguity, quota safety issues,
partial failures, delete/update edge cases, authorization boundary mistakes.
Сформулируй, каких решений в spec не хватает.
```

**Усиление:** для критичных решений берите независимую модель или живого коллегу, а не просите одну модель критиковать саму себя.

**Что сказать голосом:**

Здесь ИИ работает как въедливый аналитик. Его задача — вытащить из вас edge cases до того, как они превратятся в баги. Если исходная постановка задачи слабая — без процесса вы всё равно напишете плохой код, просто сделаете это очень быстро. Мой процесс делает слабые места видимыми до того, как вы откроете IDE.

Просить одну и ту же модель написать спеку и тут же её покритиковать — это как просить автора рецензировать собственную статью. Для серьёзных архитектурных решений берите вторую модель или, ещё лучше, живого коллегу.

## Слайд 13. PLAN: маленькие проверяемые задачи

**Большая задача почти всегда ломает LLM. Маленькая задача с явным контрактом — почти всегда помогает.**

Каждая подзадача — это контракт:

| Задача | PRE | POST | Done check |
|--------|-----|------|------------|
| CRD types | kubebuilder scaffold готов | types сгенерированы | `make generate` без ошибок |
| Namespace attachment | Project + CRD существуют | ns с label прикреплены к проекту | `status.namespaces` содержит все ns |
| Quota enforcement | Project с quotas существует | webhook отклоняет превышение | тест: pod сверх квоты → reject |
| Access reconciliation | PAB + Project существуют | RoleBinding в каждом ns проекта | тест: binding существует и корректен |

**Красный флаг:** Если план заканчивается пунктом «ну а дальше реализовать контроллер» — это не план, а иллюзия плана.

**Что сказать голосом:**

План в голове разработчика — отличная штука. Жаль только, что его нельзя заревьюить и покрыть тестами. Пока план не записан, вы не можете ограничить ИИ. Большая задача с размытыми границами гарантированно ломает контекст модели. Мы разбиваем работу на атомы не ради бюрократии, а чтобы модель не перескакивала через уровни логики и не додумывала за нас бизнес-требования.

Без явных PRE и POST условий ваш план — это просто список желаний. Контракт — это жесткая граница: что должно быть на входе, и как изменится стейт на выходе.

## Слайд 14. PLAN на примере Project Operator

Разумная последовательность шагов:

1. Scaffold проекта (kubebuilder init, cert-manager в kind)
2. Определить CRD types
3. Прогнать `make generate`
4–9. Для каждой задачи (namespace attachment, quota, access, webhooks, status):
   - сначала тесты (чистая сессия)
   - потом реализация (другая сессия)
4. Проверить покрытие, добить интеграционные тесты
5. Прогнать review и cleanup
6. Прогнать e2e / kind

**Что сказать голосом:**

Обратите внимание: тесты пишутся не в конце одним блоком, а перед каждой задачей. И обязательно в отдельной сессии — иначе AI пишет тесты под свой же код, а не под spec. Я не отдаю нейросети планирование архитектуры. Архитектурные границы я задал на шаге SPEC. Здесь ИИ работает просто как быстрый аналитик: он нарезает рутину внутри моих жестких заборов.

## Слайд 15. TEST: почему тесты отдельно от кода

**Если код уже в контексте, AI пишет тесты под реализацию, включая её баги.**

Правильный поток:

1. В **чистой сессии** попросить тесты по SPEC и PLAN
2. Человеку прочитать тесты
3. Закоммитить тесты как контракт
4. В **другой сессии** просить реализацию, которая проходит эти тесты

Формула: **tests first, implementation second, разные сессии**

**Практический нюанс:** порядок может быть и TEST→CODE, и CODE→TEST. Главное — **разные сессии**. Тот, кто пишет тесты, не должен видеть реализацию в контексте.

- TEST→CODE — безопаснее для критичных мест (webhooks, quota, access control)
- CODE→TEST — быстрее, когда спек чёткий и задача понятна

**Что сказать голосом:**

Писать тесты до кода с ИИ — это не TDD-ритуал для галочки. Это бетонный забор. Если показать модели реализацию до написания тестов, она напишет идеальные тесты, которые заботливо покроют и обоснуют все её же галлюцинации. Тесты пишутся в чистой сессии. И только потом, в другой сессии, мы просим модель в этот контракт вписаться.

Spec задаёт рамку словами. Tests задают рамку исполняемо. Поэтому TEST before CODE — это способ превратить вариативный output модели в управляемую реализацию внутри явного контракта.

## Слайд 16. Что тестировать у операторов

Минимальный набор сценариев:

- happy path
- idempotency
- partial failures
- update path
- delete path
- authorization boundaries
- admission validation
- edge cases бизнес-модели

**Красный флаг:** Если тесты получились настолько слабыми, что под них можно подогнать почти любой reconcile-loop — контракт дырявый.

**Что сказать голосом:**

Главное правило: хороший тест можно прочитать и понять, как работает система, вообще не заглядывая в код реализации. Если тесты получились такими водянистыми, что под них можно подогнать любой кривой reconcile-loop — значит, ваш контракт дырявый. Идите и переписывайте тесты, пока к коду даже не притрагивались.

Одна из самых частых ошибок — написать красивые тесты на моки, которые почти ничего не говорят о корректности reconcile-поведения.

## Слайд 17. CODE: реализация зафиксированного контракта

CODE — это не «пусть AI напишет решение». Это «пусть AI реализует уже зафиксированный контракт».

**Input:** spec, implementation plan, конкретная задача из PLAN, **тесты, которые уже написаны и падают**, project context.

**Выход:** все тесты проходят, `make build` и `go vet` проходят, пробелы в spec зафиксированы, а не решены молча.

**Красный флаг:** Если вы не можете объяснить, почему код написан именно так — вы потеряли контроль.

**Что сказать голосом:**

Да, на этом этапе ИИ — это просто очень быстрый template expander. И в этом нет проблемы. Инженерное открытие никуда не исчезло, оно просто сместилось влево: в архитектуру, декомпозицию и контракты. Скорость печати символов дает модель, а управляемость этой скорости даете вы.

Зеленые тесты — это еще не повод пушить в мейн. Код считается хорошим, только если вы можете объяснить, почему каждая строчка написана именно так. В промпте нет никаких магических слов в стиле «будь сеньор-разработчиком». Есть жесткая ссылка на спеку, падающие тесты и лимит по строкам.

## Слайд 18. REVIEW: Reviewer ≠ Implementer

Генерировать код и ревьюить его в одном окне чата — как просить студента самому себе поставить оценку.

**Рабочая схема:**

- Terminal 1 / Session 1 — `Implementer`
- Terminal 2 / Session 2 — `Reviewer`

**Как реализовать:** `git worktree` — физическое разделение контекста.

**Реальные примеры из review нашего оператора:**

1. **Лишняя абстракция.** AI обернул `rules` в `ProjectRoleSpec`, хотя нативные `Role`/`ClusterRole` держат rules на верхнем уровне. AI следовал шаблону kubebuilder, а не convention Kubernetes API.

2. **Генерация «на вырост».** AI создал 4 фазы `ProjectNamespacePhase` (Pending, Active, Terminating, Failed), но в коде реально использовалась только Active. Мусор, который усложняет API без причины.

3. **Расхождение кода и документации.** Комментарий к `AllowDeletion` описывал удаление namespace, а реализация блокировала удаление ресурса `ProjectNamespace` через webhook.

Ни одну из этих проблем нельзя поймать линтером или тестом. Только человек с пониманием модели.

**Красный флаг:** Если review заканчивается фразой «в целом выглядит нормально» — это не review.

**Что сказать голосом:**

Генерировать код и ревьюить его в одном окне чата — это как просить студента самому себе поставить оценку за диплом. У нас это физически разные сессии. Модель-ревьюер не независима, но это недорогой и полезный второй проход по контракту. Он помогает поймать сломанный RBAC и расхождение бизнес-логики до PR.

Вот эти три примера на слайде — чисто семантические, и без живого ревьюера они бы уехали в прод. Если ревьюер пишет вам «в целом выглядит нормально» — это не ревью. Это отписка. Хорошее ревью всегда пытается сломать вашу логику об исходную спеку и тесты.

## Слайд 19. Что автоматизировать, а что нет

**Автоматизировать (hooks):**

- форматирование, линтер, `go vet`
- unit/envtest на коммит
- запрет опасных команд
- stop-checklist перед завершением задачи

**Оставить человеку:**

- бизнес-логика и семантика
- качество выбранного решения
- архитектурные компромиссы
- correctness и safety

**Gotcha:** инструменты вроде gopls не успевают за скоростью генерации. Линтер и vet лучше запускать на коммит, а не на каждое сохранение.

**Что сказать голосом:**

Hooks не делают код правильным. Они делают ревью менее шумным. Автоматизировать нужно всё, что проверяется дешево: линтеры, форматирование, компиляцию. Но hooks принципиально не отвечают на вопросы бизнес-логики. Всё, что требует инженерного суждения, остается на человеке.

## Слайд 20. Project context file и LEARN

**Контекст — это не «вот тебе весь репозиторий». Контекст — это сжатая инженерная память проекта.**

Что хранить в project context file (в нашем случае `CLAUDE.md`):

```text
# Commands
make generate / make test / go vet / golangci-lint run

# Spec
See SPEC.md for full model, invariants, and edge cases

# Known Gotchas
Don't use r.Get() after r.Update() without re-fetch
meta.SetStatusCondition() needs observedGeneration
Deleting one namespace must not break full project reconciliation
Webhook must be fast — read status, don't list pods
```

**Четыре категории:** команды сборки, инварианты модели, known gotchas, review checklist.

**Почему это не помойка:** context file лежит в репозитории, версионируется и проходит review через PR. Каждое правило заработано — оно появляется, потому что без него реально ломалось. Если правило устарело — удаляется на review, как deprecated код.

**Красный флаг:** Если через неделю вы снова объясняете модели одни и те же базовые правила проекта — этап LEARN провален.

**Что сказать голосом:**

Если скормить модели весь репозиторий, она начинает хуже держать нужные абстракции. Вместо этого мы даем ей project context file — в нашем случае это `CLAUDE.md`, но название не принципиально, у Cursor это `.cursorrules`, у Copilot — `copilot-instructions.md`. Контекст — это не дамп исходников. Это сжатая инженерная память проекта: жесткие конвенции, команды сборки и список граблей, на которые не надо наступать дважды.

Project context file — это ваша защита от амнезии ИИ. Если через неделю вы снова объясняете модели одни и те же базовые правила, значит LEARN не сработал. Наша цель — сделать так, чтобы каждая следующая задача в этом репозитории делалась быстрее и чище.

## Слайд 21. Вся цепочка как pipeline артефактов

**SPEC produces:** model, invariants, edge cases, acceptance criteria

**PLAN consumes SPEC and produces:** tasks, dependencies, contracts

**TEST consumes SPEC + PLAN and produces:** executable contract (тесты падают — реализации ещё нет)

**CODE consumes PLAN + TEST and produces:** реализация, которая проходит тесты

**REVIEW consumes SPEC + TEST + DIFF and produces:** fixes / decisions

**LEARN consumes everything and produces:** reusable project memory

**Что сказать голосом:**

Вот в этом месте я бы хотел, чтобы вы перестали думать про LLM как про чат и начали думать про него как про участника pipeline.

Каждый этап ловит свой класс ошибок. Дыры в модели всплывают на SPEC. Невозможность выразить expected behavior — на TEST. Расползание scope — на CODE, потому что diff маленький. Semantic bugs — на REVIEW, потому что reviewer проверяет против spec и тестов. Я не убираю риск ошибки. Я двигаю обнаружение ошибки как можно левее по pipeline.

## Слайд 23. На каких задачах этот подход работает

Лучшие кандидаты:

- Kubernetes controllers and operators
- internal platform tooling
- API / CRD driven systems
- backend задачи с явной моделью и инвариантами
- refactoring при хорошем test harness

Менее удачные первые кандидаты:

- размытые product-фичи без чёткой модели
- большие cross-cutting refactors без границ
- задачи, где вы сами ещё не понимаете desired behavior
- «перепиши полрепозитория»

**Что сказать голосом:**

Давайте честно, не тащите этот фреймворк чинить опечатку в README или писать скрипт на 50 строк. Он вам там не нужен. Этот процесс окупается там, где когнитивная нагрузка и так уже пробивает потолок: новые доменные модели, хитрый стейт-менеджмент, квоты и доступы. Мы не добавляем оверхед, мы берем уже существующий хаос сложной задачи и делаем его управляемым.

## Слайд 24. Анти-паттерны

- «Сделай всё сразу»
- «Вот тебе весь контекст мира»
- AI пишет и тесты, и реализацию без разделения
- один и тот же инстанс и пишет, и ревьюит
- большие diff на сотни строк
- review отсутствует
- known gotchas нигде не записаны
- после трёх провалов человек не вмешивается и продолжает долбить тот же prompt

**Реальный пример:**

AI сгенерировал `ProjectRole` со стандартной kubebuilder-структурой `Spec`/`Status`, хотя по API convention Kubernetes `rules` должны быть top-level. Выглядит профессионально, компилируется, тесты проходят — но это неправильный API design.

**Короткая формула:** Плохой процесс + AI = быстрый путь к плохому результату.

**Что сказать голосом:**

Плохой инженерный процесс плюс быстрый ИИ ускоряет выход плохих решений в production. AI может сгенерировать внешне аккуратный, но неверный API design — `ProjectRole` с нестандартными структурами, всё компилируется, тесты зеленые. Без человека, который понимает нативные API-конвенции Kubernetes, это просочится в релиз. LLM не убирает необходимость инженерной фильтрации, а повышает цену ошибки.

Граница простая: как только output дешевле сгенерировать, чем качественно отревьюить — LLM перестаёт помогать. Мой процесс — small diffs, separate review, tests before code — как раз нужен, чтобы не переходить эту границу.

## Слайд 25. Что делать завтра в своей команде

1. Выделить одну реальную инженерную задачу с чёткой моделью
2. Перед кодом написать короткий spec
3. Разбить задачу на маленькие контракты с критерием готовности
4. Завести project context file (`CLAUDE.md`, `.cursorrules` или аналог)
5. Развести implementer / tester / reviewer роли
6. После каждой задачи обновлять project memory

**Что сказать голосом:**

И тут важный вопрос: «А как это внедрять в команде из пяти-семи человек, чтобы процесс не превратился в бюрократический театр?» Ответ: не внедрять как полный фреймворк на все задачи. Командное внедрение для меня — это минимальный набор артефактов и ротация ролей на сложных задачах. Первое: выбрать один класс задач, где процесс реально нужен — контроллеры, операторная логика, сложные backend-фичи с инвариантами. Не тащить это на мелкие багфиксы. Второе: минимальные артефакты — короткий spec, plan с маленькими задачами, тестовый контракт до реализации, отдельный review. Третье: лёгкая ротация ролей — один пишет spec, другой его читает, третий ревьюит план или тестовый контракт. Автор кода не является единственным reviewer своего diff.

## Слайд 26. Главный вывод

- LLM не заменяет инженерный процесс
- LLM усиливает инженерный процесс
- Чем сложнее система, тем важнее: spec, декомпозиция, тестовый контракт, review, project memory
- Главная единица успеха — не prompt
- Главная единица успеха — **управляемый цикл разработки**

**Финальная фраза:**

**Не AI пишет код. Инженер пишет код руками AI.**

**Что сказать голосом:**

Цена рутины упала до нуля, а цена архитектурной ошибки взлетела в космос. Фреймворк, который я показал — это способ управлять этой новой сложностью. Инженер не стал придатком к промпту. Наоборот. Теперь вы можете удерживать архитектуру целого куска платформы в одиночку, потому что процесс делает вашу экспертизу масштабируемой. Не AI пишет код. Инженер пишет код руками AI.

Даже если завтра выйдет модель в пять раз умнее, процесс станет только важнее. Чем мощнее инструмент, тем дороже обходится хаотичная работа с ним. Мой доклад не про workaround под слабые LLM, а про безопасное встраивание сильных LLM в инженерную работу.

## Слайд 27. Чек-лист: сфотографируйте этот слайд

```
SPEC:  ✓ Инварианты записаны
       ✓ Edge cases зафиксированы
       ✓ Два инженера одинаково понимают desired behavior

PLAN:  ✓ Каждая задача имеет PRE/POST conditions
       ✓ Критерий готовности для каждого шага
       ✓ Нет задач «ну и дальше реализовать остальное»

TEST:  ✓ Тесты в отдельной сессии, без кода в контексте
       ✓ Тесты проверяют модель, а не реализацию
       ✓ Есть unhappy paths и destructive scenarios

CODE:  ✓ Пробелы в spec фиксируются, а не решаются молча
       ✓ make build + go vet проходят
       ✓ Вы понимаете каждую строку

REVIEW:✓ Reviewer ≠ implementer
       ✓ Проверка по spec, а не «выглядит нормально»
       ✓ Semantic bugs, а не только стиль

LEARN: ✓ Project context file обновлён
       ✓ Gotchas записаны
       ✓ Следующая сессия сильнее предыдущей
```

**Начните с трёх вещей:** напишите spec, разбейте задачу на маленькие контракты, не давайте одной и той же сессии одновременно быть автором, тестировщиком и ревьюером. Этого уже достаточно, чтобы качество работы с LLM резко изменилось.

**Что сказать голосом:**

Самый частый способ сломать этот метод — начать срезать углы. Если одна и та же сессия пишет код, тесты и ревью, границы этапов размываются, и процесс перестаёт давать контроль. Держите границы этапов, и ИИ будет работать на вас.

Этот чек-лист полезен не только когда процесс работает, но и когда он ломается. Без процесса команда просто говорит «LLM какая-то плохая». С процессом можно диагностировать конкретный сломанный этап.
