# Дисклеймер

Вчера я рассказал на DevOpsConf, как написал production Kubernetes-оператор за 7 дней вместо 90.

Люди спросили: **«А можно это повторить, не будучи тобой?»**

Сегодня я покажу, как он работает. С живой записью терминала. И честно разберу, где Claude Code облажался — а потом покажу, как правильный workflow это бы предотвратил.

---

<!-- slide 3 -->

# Краткое содержание вчерашнего доклада

> 4 CRD · 6 reconciler'ов · 7 webhook'ов · 5 PR
> Медиана PR — 155 строк · Тестов в 3.6× больше кода

Я использовал **процесс**:

```
SPEC → PLAN → TEST → CODE → REVIEW → LEARN
```

Каждый этап — точка решения **человеком**. AI не двигается дальше без accept.

Проблема: этот процесс жил у меня в голове. А голова — плохой носитель для воспроизводимых результатов.

<!-- slide 4 -->

# Как выглядит жизнь без процесса

```
Вы:  «Сделай мне Kubernetes-оператор»
AI:  *пишет 2000 строк за 3 минуты*
Вы:  «Ого, быстро!»
AI:  *молча удалил owner references, потому что так проще*
Вы:  *3 дня дебажит*
```

Раньше мы плодили баги неделями.
Теперь можем генерировать их **тысячами строк в секунду**.

Проблема не в LLM. Проблема в том, что вы дали автомат Калашникова обезьяне и удивляетесь, что она стреляет не туда.

# MAP Framework — что это

**M**odular **A**gentic **P**lanner. Open-source CLI для Claude Code.

```bash
pip install mapify-cli
cd your-project && mapify init
claude
> /map-efficient implement user auth with JWT
```

**11 специализированных агентов**, state machine, branch-scoped артефакты. Не один промпт на все случаи жизни, а конвейер с quality gates.

Сегодня я покажу, что бывает, когда выбираешь **неправильный** workflow — и как правильный спас бы ситуацию.

> github.com/azalio/map-framework

---

<!-- slide 6 -->

# Два workflow — две философии

| | `/map-efficient` | `/map-plan` → `/map-tdd` → `/map-task` |
|---|---|---|
| **Тесты** | После кода, в той же сессии | ДО кода, в отдельной фазе |
| **Monitor** | Должен быть субагентом... но AI может схалтурить | Enforcement через TEST_FAIL_GATE |
| **Контроль** | AI сам решает порядок | Человек подтверждает каждый шаг |
| **Скорость** | Быстро | Медленнее, но надёжнее |
| **Когда** | Низкий риск, хороший test harness | Сложные системы, новые домены |

Я запустил `/map-efficient` на production-задаче. Давайте посмотрим, что получилось.

<!-- slide 7: ДЕМО -->

# ДЕМО: /map-efficient — что пошло не так

> Запись терминала (asciinema). Реальный проект. Реальные ошибки.

**Задача:** `add health check endpoint with readiness and liveness probes`

**17 минут, 8 подзадач, 23 теста, PR готов.**

Звучит круто, да? А теперь давайте разберём, что пошло не так.

---

<!-- slide 8 -->

# Баг №1: Амнезия в одной сессии

```
python3 .map/scripts/map_orchestrator.py validate_step "2.3"
→ Error: No such file '.../project-operator/.map/scripts/...'
```

Claude запускал orchestrator из `project-operator/`, а скрипты лежат в корне репо.

**Ошибся 4 раза. На ST-001, ST-002, ST-006, ST-007.**

Каждый раз «удивлялся» заново. Не запомнил собственный фикс из предыдущего шага.

С `/map-task` — каждая подзадача в отдельной сессии. Нет накопления мусора в контексте. Но и нет амнезии — потому что state лежит в `.map/`, а не в голове у AI.

---

<!-- slide 9 -->

# Баг №2: Тесты видели реализацию

Это **главный** косяк.

```
ST-002: Actor пишет readiness.go
ST-004: Actor пишет readiness_test.go    ← в той же сессии!
```

Тесты видели реализацию. AI заботливо покрыл свой собственный код, включая потенциальные баги.

**Как `/map-tdd` это предотвращает:**

```
Phase: TEST_WRITER  → пишет тесты ИЗ СПЕКИ (не видит реализации)
Phase: TEST_FAIL_GATE → тесты ДОЛЖНЫ ПАДАТЬ (иначе — назад)
Phase: ACTOR → пишет код, чтобы тесты прошли
```

Тесты пишутся из контракта (AAG), а не из кода. Red → Green → Refactor. Не «Green из коробки, потому что я подглядел».

---

<!-- slide 10 -->

# Баг №3: Monitor — самозванец

По спеке `/map-efficient`, Monitor — это **отдельный агент**:

```python
Task(subagent_type="monitor", prompt="Validate written code...")
```

В демо Claude **сам «играл роль»** Monitor. Просто написал:

> «Monitor pass — VC1, VC2, VC3»

...после `go vet` и `go build`. Без отдельного субагента. Без проверки SPEC.md. Без анализа race conditions. Без проверки graceful shutdown.

**Студент поставил себе пятёрку.**

С `/map-task` — Monitor вызывается автоматически как субагент. У него нет доступа к контексту Actor'а. Он не «знает», что код «должен быть правильным».

---

<!-- slide 11 -->

# Баг №4: Мелочи, которые складываются

**`go build ./cmd/...`** — упал, потому что `cmd` — директория:

```
→ Error: build output "cmd" already exists and is a directory
```

CLAUDE.md не описывал правильную команду. Фикс: `go build -o /dev/null ./cmd/main.go`

**`import yaml`** — не установлен:

```
→ ModuleNotFoundError: No module named 'yaml'
```

Claude пытался валидировать YAML без PyYAML. Пришлось изобретать обходной путь.

**Каждая мелочь — лишние 30 секунд и расход контекста.** А контекст — это самый дорогой ресурс при работе с LLM.

---

<!-- slide 12 -->

# Как должно было быть: /map-plan → /map-tdd → /map-task

```bash
# Шаг 1: Декомпозиция (человек ревьюит план)
/map-plan add health check probes

# Шаг 2: Тесты ДО кода (для каждой подзадачи)
/map-tdd ST-002    # пишет тесты для CacheReadinessChecker
                   # TEST_FAIL_GATE проверяет: тесты падают? Хорошо.

# Шаг 3: Реализация (в новой сессии, под тесты)
/map-task ST-002   # Actor пишет код → Monitor (субагент!) валидирует
                   # Тесты зелёные? Тогда дальше.

# Повторить для ST-003, ST-004...

# Шаг 4: Финальная проверка
/map-check
/map-review
```

Каждая подзадача — изолированная сессия. Тесты не видят реализацию. Monitor — настоящий субагент.

# Итого: /map-efficient vs правильный flow

| Проблема в демо | `/map-efficient` | `/map-plan` → `/map-tdd` → `/map-task` |
|---|---|---|
| Тесты видели реализацию | Написаны после кода | TEST_WRITER → TEST_FAIL_GATE → ACTOR |
| Monitor — самозванец | AI сам себя проверил | Отдельный субагент, без контекста Actor |
| Амнезия (потерял pwd) | Один длинный контекст | Каждая задача — чистая сессия |
| Накопление мусора | 18k+ токенов к концу | ~3-5k на подзадачу |

**`/map-efficient` — это удобно. Но для production-кода используйте полный flow.**

Быстро ≠ правильно. А «дорого» в заголовке — это не про деньги. Это про цену ошибки.

<!-- slide 14 -->

# LEARN: что я починю после этого демо

**1. CLAUDE.md — добавить правильные go build команды**
→ Known Gotchas: `go build -o /dev/null ./cmd/main.go`, не `./cmd/...`

**2. Orchestrator guard — проверка pwd**
→ Хук, который ловит вызов из неправильной директории

**3. `/map-efficient` без `--tdd` — запретить писать тесты**
→ Если в плане есть test-подзадачи, принудительно включать TDD-фазы

**4. Monitor enforcement**
→ Workflow невалиден без `Task(subagent_type="monitor")`

Каждый прогон делает следующий чище. Баги фреймворка — это тоже данные.

# Что делать в понедельник

```bash
# Попробовать MAP
pip install mapify-cli
cd your-project && mapify init

# Для production-задач — полный flow:
/map-plan <задача>
/map-tdd ST-001
/map-task ST-001
/map-tdd ST-002
/map-task ST-002
...
/map-check
/map-review
```

Для быстрых задач с низким риском — `/map-efficient`.
Для всего остального — **`/map-plan` → `/map-tdd` → `/map-task`**.

# Главный вывод

> **Не AI пишет код. Инженер пишет код руками AI.**

Вы видели в демо: даже с фреймворком AI косячит. Но правильный workflow превращает невидимые баги в видимые, а неуправляемый хаос — в конвейер с чекпоинтами.

`/map-efficient` — быстро, но поверхностно.
`/map-tdd` + `/map-task` — медленнее, но каждый баг ловится на своём этапе.

**Выбор workflow — это и есть инженерное решение.**
